# v0.11.0 Release Notes

**Date:** 2026-04-17

Minor version bump. v0.11.0 replaces the file-based `open-work.md` work-item tracker with a pluggable API adapter. Issues live in an external system (GitHub Issues by default) and are reached through a thin interface at `.claude/scripts/trackers/interface.mjs`. The session-integration skills (`/start-work`, `/pause-work`, `/complete-work`, `/setup-tracker`, `/workspace-init`) now talk to the adapter instead of reading and writing a markdown file. Assignment becomes atomic — the concurrent-pickup race the old design had is structurally gone.

## Features

### Tracker adapter interface

New module at `template/.claude/scripts/trackers/interface.mjs` exposes `createTracker(config)` and an `AlreadyAssignedError` class. Skills import only from here; concrete backends live as sibling modules selected by `workspace.tracker.type`. The interface is eight methods: `listAssignedToMe`, `listUnassigned`, `getIssue`, `claim`, `createIssue`, `comment`, `closeIssue`, `ensureLabels`, plus `ensureMilestone` for idempotent milestone provisioning. Issue IDs are adapter-prefixed opaque strings (`gh:42`) so session trackers are self-describing across backend swaps.

### GitHub Issues adapter

`template/.claude/scripts/trackers/github-issues.mjs` is the first shipped backend. It wraps the `gh` CLI via an injectable `spawnFn` — the same pattern the session-layout migrator uses — so tests run with canned mock responses and zero external dependencies. Twelve unit tests cover normalization, the atomic-claim concurrency guard (`AlreadyAssignedError` when an issue is already assigned to another user), idempotency of `claim` when the caller already owns the issue, and `gh` failure surfacing. Config is minimal:

```json
{
  "workspace": {
    "tracker": {
      "type": "github-issues",
      "repo": "your-org/your-workspace"
    }
  }
}
```

`repo: "auto"` resolves to the workspace's own git remote. The workspace repo is the default target — no separate `workspace-{project}` repo is created; the scaffolder assumes the workspace repo itself hosts the issues.

### Atomic claim on `/start-work`

The blank flow fetches issues assigned to the current user first. If that list is empty, it falls back to all unassigned issues. When the user picks from the fallback list, `/start-work` calls `adapter.claim(issueId)` — which GETs the issue, checks that no other user holds it, and `gh issue edit --add-assignee` in one step. If another user grabbed it between the list fetch and the claim, the adapter throws `AlreadyAssignedError` with the conflicting user's handle; the skill surfaces the collision and re-fetches. The race the old file-based design had — two teammates seeing the same `open` ticket and starting parallel work — is structurally closed. Assignment is the lock.

### Close-on-merge in `/complete-work`

After all PRs land in Step 9, Step 10 calls `adapter.closeIssue(workItem, { comment })`. The comment lists merged PR URLs and a synthesis of the release notes just written. If the tracker is unreachable at close time, the error surfaces in the unified summary but cleanup proceeds — no data is at risk. Blank sessions (no `workItem:` linkage) skip the close silently.

### Pause comment in `/pause-work`

Step 3 now posts the session's `## Progress` body as a comment on the linked issue via `adapter.comment()` instead of editing a local `open-work.md`. The tracker has no native "paused" concept, so the comment plus continued assignment is the signal.

### `/setup-tracker` rewritten

Single interactive flow: pick a backend (GitHub Issues is the only one shipped), confirm `gh auth`, resolve the target repo (defaults to the workspace remote), write the `tracker` block to `workspace.json`, call `adapter.ensureLabels()` to create the six standard labels (`bug`, `feat`, `chore`, `P1`, `P2`, `P3`), optionally create starter milestones via `adapter.ensureMilestone()`. Both `ensureLabels` and `ensureMilestone` are idempotent, so re-running setup never duplicates.

### `/workspace-init` drops open-work.md population

Step 12 (populate `open-work.md` from discovered items) is gone. The new Step 12 prompts to run `/setup-tracker` and surfaces any candidate work items found during init without auto-creating issues — the user triages them via `/start-work → Something new`. Subsequent steps renumber accordingly.

### Migration script for existing workspaces

`template/.claude/scripts/migrate-open-work.mjs` is a one-shot utility that parses an existing `shared-context/open-work.md` and creates real issues via the configured adapter. For each ticket it prints the source-to-issue-ID mapping. Not idempotent — if it fails partway, the operator cleans up orphan issues manually and re-runs. Used successfully during the dogfood migration of this workspace: three tickets on the `app-vision-rewrite` session branch became issues #20, #21, #22 on `ukt-solutions/ulysses-workspace`.

### New rule: `work-item-tracking.md`

Replaces `open-work-tracking.md`. The new rule is tracker-agnostic — it describes the adapter interface, the `workItem:` session-linkage convention, when to create issues during work, and what never to do (no local mirrors, no caching issue bodies, no writing state anywhere except the tracker).

## Maintenance

- `template/.claude/scripts/sync-open-work.mjs.reference` removed. Superseded entirely by the adapter; the conceptual "sync" step no longer exists because the tracker is the source of truth, not a presentation layer.
- `template/.claude/rules/open-work-tracking.md` removed. Replaced by `work-item-tracking.md`.
- Session tracker frontmatter's `workItem:` field now holds adapter-prefixed IDs (`gh:42`) instead of internal integer IDs. The prefix identifies the backend so frontmatter is self-describing across adapter changes.

## Breaking changes

- **`shared-context/open-work.md` is no longer maintained by any skill.** Existing workspaces should run the migration script once, then delete the file. Any references to `open-work.md` in custom rules, scripts, or documentation need to be removed or retargeted to the adapter.
- **`workspace.json → tracker.sync` is ignored.** The v0.10 schema had `{ type, sync, defaultMilestone }`; the new schema is `{ type, repo }`. The adapter path is conventional (`.claude/scripts/trackers/{type}.mjs`), not configured. Running `/setup-tracker` after upgrade rewrites the block.
- **Session tracker `workItem:` values.** Old sessions with integer `workItem:` values won't resolve through the new adapter. Upgrade by opening the corresponding issue on GitHub and replacing the frontmatter value with `gh:N`.
- **`templateVersion` jumps to 0.11.0.** Workspaces running 0.10.x need to `npx create-ulysses-workspace --upgrade` then `/workspace-update` to apply the template changes.

## Upgrade guide for existing workspaces

1. `npx create-ulysses-workspace --upgrade` — stages the v0.11 payload.
2. `/workspace-update` — applies the new `.claude/*` (adds `trackers/`, migration script, new rule; removes old rule and sync script).
3. If the workspace has `shared-context/open-work.md` with tickets:
   - Run `/setup-tracker` to write the new `tracker` block and initialize labels.
   - Run `node .claude/scripts/migrate-open-work.mjs shared-context/open-work.md workspace.json` from the workspace root.
4. Delete `shared-context/open-work.md` and commit.
5. Any session trackers with integer `workItem:` values: update by hand with the new `gh:N` form.

## What does not change

- The session tracker frontmatter parser (`session-frontmatter.mjs`) — all fields and formats unchanged.
- `/handoff`, `/braindump`, `/promote`, `/release`, `/sync-work`, `/maintenance`, `/workspace-update` — none were tracker-aware to begin with.
- Workspace worktree and project worktree nesting.
- Cleanup and teardown order in `cleanup-work-session.mjs`.

## Tests

- `template/.claude/scripts/trackers/interface.test.mjs` — 4 assertions covering factory behavior, error class shape, and lazy adapter construction.
- `template/.claude/scripts/trackers/github-issues.test.mjs` — 11 assertions covering normalization, concurrency semantics of `claim`, `ensureMilestone` idempotency, and `gh` failure surfacing.

Both run via plain `node path/to/file.test.mjs`.

## Dogfood verification

Before merge, the adapter was exercised live against `ukt-solutions/ulysses-workspace`: six standard labels created via `ensureLabels()`, `Backlog` and `v0.12 — Tracker abstraction` milestones created via `ensureMilestone()`, three tickets migrated from an existing `open-work.md` via the migration script, fifteen new tickets seeded from the product inventory, and the close-on-merge flow shut down the tracker-setup ticket (`gh:21`) when this PR merged. All happy-path operations passed. Twelve unit tests pass.
