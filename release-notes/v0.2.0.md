# v0.2.0 Release Notes

**Date:** 2026-04-03

## Features

### Non-interactive payload delivery

The CLI no longer writes template files directly to the workspace or prompts interactively for file conflicts. Running `--init` or `--upgrade` stages the full template into a `.workspace-update/` directory and exits. Skills handle all interactive application — diffing files, asking before overwriting, merging settings — on the user's next Claude Code session.

**New CLI flags:**
- `npx create-claude-workspace --init [dir]` — stages the template payload, creates a bare `workspace.json`, wires a bootstrap hook, and adds temporary `@` imports to CLAUDE.md for mandatory rules during initialization. Non-interactive and idempotent.
- `npx create-claude-workspace --upgrade [dir]` — verifies the workspace is initialized, stages the payload with version transition info, and exits. Errors with a pointer to `--init` if the workspace isn't initialized.
- `npx create-claude-workspace --migrate` — deprecated. Prints an error directing users to `--init` or `--upgrade`.

### Automatic bootstrap via hooks

A new `workspace-update-check.mjs` hook runs on both SessionStart and PreToolUse events. It checks for a `.workspace-update/` payload directory and instructs Claude to read and follow the appropriate skill (`/workspace-init` or `/workspace-update`) before doing anything else. This eliminates the previous gap where users had to read terminal output and manually invoke a skill — the workspace bootstraps automatically on the next interaction.

### Cross-platform Node.js hooks

All template hooks converted from bash scripts to Node.js modules (`.mjs`). Node.js is already a prerequisite for the CLI, so this adds no new dependencies while enabling Windows support. A shared utility module (`_utils.mjs`) provides common patterns: reading stdin JSON, resolving the workspace root, reading JSON config files, and emitting JSON responses. This replaces the previous dependency chain of bash + python3 + jq.

Hooks converted: session-start, session-end, pre-compact, post-compact, subagent-start, repo-write-detection, worktree-create. New hook added: workspace-update-check.

### Updated skills

Both `/workspace-init` and `/workspace-update` now read from the `.workspace-update/` payload directory instead of assuming files are already installed. They move components from the payload to their final locations with user approval at each step, then delete the payload directory. The `/workspace-update` skill also handles the one-time `.sh` → `.mjs` hook migration for workspaces upgrading from pre-0.2.0.

## Maintenance

### Module restructure

Replaced the monolithic `lib/migrate.mjs` (318 lines, mixed concerns) with three focused modules:
- `lib/payload.mjs` (~40 lines) — shared payload staging used by both init and upgrade
- `lib/init.mjs` (~60 lines) — fresh install wiring: payload, workspace.json, hook, CLAUDE.md
- `lib/upgrade.mjs` (~20 lines) — upgrade staging with initialization verification

## Known Limitations

- **Hook additionalContext reliability** — the bootstrap mechanism relies on Claude following MANDATORY instructions in `additionalContext`. Works in practice but isn't enforced by the platform.
- **PreToolUse hook overhead** — the workspace-update-check hook fires on every tool call. Bails fast with a single `existsSync` check when no payload exists, but the per-call overhead is non-zero.
- **Node.js runtime requirement** — hooks now require Node.js at runtime, not just install time. Safe to assume Node.js is present if the CLI was run via npx.
- **Pre-flag workspace compatibility** — workspaces created before the `initialized` flag don't have `initialized: true`. The `--upgrade` command accepts `templateVersion` as a fallback, but `/workspace-init`'s gate check may still need the same fallback.
