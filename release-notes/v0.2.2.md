# v0.2.2 Release Notes

**Date:** 2026-04-05

## Fixes

Fixed design debt in the template — contradictions between rules and skills, an undefined placeholder in git commands, and a stale spec location instruction. No new capabilities.

**Rule-skill separation.** Three rules contained workflow instructions that belong in skills. `git-conventions.md` had commit timing policy ("do not commit unless asked") and shared-context bundling instructions. `memory-guidance.md` had skill invocation triggers ("suggest /braindump", "flag 20+ turns without capture"). `workspace-structure.md` had a spec lifecycle instruction ("specs start in inflight, move to worktree"). All three rules were trimmed to conventions and principles only — the workflow behavior they described is already handled by the skills that execute it.

**Override acknowledgments.** Three skills — handoff, braindump, and complete-work — auto-commit files without user request, which contradicted the git-conventions rule. Each skill's Notes section now explicitly documents the override. Handoff and braindump clarify they bypass the "do not commit unless asked" convention (but not the "committed individually" constraint). Complete-work notes that context consumption, cleanup, and release note commits are intentional workflow behavior.

**Branch field clarification.** The `start-work` and `complete-work` skills used `{default-branch}` as an undefined placeholder in git commands. Both now explicitly reference `repos.{repo}.branch` from workspace.json, closing the gap between what's declared in config and what skills actually use.

## Known Issues

- The workflow instructions removed from rules (context capture triggers, session awareness, commit timing) are not currently codified anywhere else. They existed as ambient guidance. Whether any should be formalized as dedicated skills or hooks remains open. The PreCompact/PostCompact capture logic is already tracked as inventory items #15 and #16.

- The `{repo-branch}` placeholder in skills now points to workspace.json, but nothing validates that the field matches the actual remote default branch. If a repo's default branch changes on GitHub, workspace.json becomes stale silently. A `/maintenance audit` check could catch this.
