# v0.6.0 Release Notes

**Date:** 2026-04-12

Minor version bump — this release folds several accumulated features into a single release document, adds the milestone-aware work tracking system, and completes the rename to `create-ulysses-workspace`.

## Features

### Milestone-aware open work tracking

The `open-work-tracking` rule and surrounding skills now support milestones as a first-class concept. The `open-work.md` table format is now 8 columns (added Milestone between Pri and Status), and a new details section convention (`### #N — Title`) lets each ticket carry full root-cause analysis, file paths, and proposed approach alongside the scannable table.

Milestones use a short-form convention: the sync script takes the first whitespace-delimited token of each milestone's title as its short form, so `"v0.1 — Alpha"` becomes `v0.1` in the table. This keeps tables compact while milestone titles carry themes, dates, and narrative context.

A new `defaultMilestone` field in `workspace.json`'s tracker config assigns unmilestoned tickets to a fallback (typically `Backlog`) at sync time.

### GitHub Issues sync — reference implementation ships with the template

`template/.claude/scripts/sync-open-work.mjs.reference` is a ~300-line working GitHub sync script covering label creation, milestone resolution, tracker-state block management, and crash safety. The `/setup-tracker` skill now has a fast-path for GitHub workspaces: copy the reference, run `--init-labels`, create milestones, configure, done. For other trackers (Linear, Jira, Notion), `/setup-tracker` still walks through building a custom sync script using the reference's parsing logic as a starting point.

### Open work tracking — workspace-level by default

Tickets live in the workspace repo's tracker, not individual project repos. Cross-cutting work doesn't split across multiple trackers. Release milestones span the whole product. One source of truth.

### Start-work integrates with open-work

`/start-work` presents open items grouped by milestone (highest priority first within each group) when starting a new session. Picking an existing item links the session to it and transitions the item to `in-progress`. Describing something new adds a fresh item with the next ID. The session marker gets a `workItem` field so `/pause-work` and `/complete-work` can update the linked item's status automatically.

### Standalone /setup-tracker skill

Tracker configuration is no longer trapped inside `/workspace-init`. `/setup-tracker` can be invoked any time mid-project when someone decides they need issue tracking. `/workspace-init` Step 13 now delegates to it rather than inlining the logic.

### Workspace-level issue tracking (from v0.5.0)

Previously released as internal v0.5.0 but never written up:
- `open-work-tracking` rule defines the table format, update triggers, and sync conventions
- `/start-work`, `/pause-work`, `/complete-work` wire up automatic status transitions
- `workspace.json` gains a `tracker` config field

## Fixes

### CRITICAL: .gitignore symlink destruction bug

Fixed a destructive bug in the template's `.gitignore` that could silently destroy a user's entire `repos/` directory — project clones, worktrees, active work sessions.

**What was happening:** The start-work helper creates a symlink named `repos` inside every workspace worktree (so the worktree can see its sibling project worktrees). The template's `.gitignore` had `repos/` with a trailing slash. Git's gitignore patterns treat trailing-slash patterns as matching real directories only, not symlinks. So the symlink was NOT being ignored. Running `git add -A` inside a worktree staged the symlink. Once merged to main and pulled at the workspace root, git replaced the real `repos/` directory with the symlink, destroying everything inside.

**Three layers of protection in the fix:**

1. Template `.gitignore` updated. `repos/` and `.claude-scratchpad/` lose their trailing slashes — patterns now match both directories and symlinks. A detailed comment explains the reasoning.
2. `workspace-update` skill adds a migration step (Step 2b) that runs before any other update. It checks for the vulnerable pattern and offers to fix it, including untracking any `repos` symlink that was already committed.
3. `create-work-session.mjs` refuses to run with the vulnerable pattern. Belt-and-suspenders — prevents the bug even if a user skips `workspace-update`.

Existing workspaces are migrated automatically when they run `/workspace-update`.

### Package.json version sync (from v0.4.2)

`/release` now bumps `package.json` as part of Step 7b, preventing the version drift that previously left the CLI displaying stale template version numbers during `--upgrade`.

### /start-work captures prior brainstorming (from v0.4.1)

Brainstorming, spec writing, and design discussion that happens before `/start-work` is now detected and captured into the inflight tracker, so `/complete-work` produces release notes with full design context instead of thin summaries.

### /complete-work cleans up inflight tracker (from v0.4.1)

Step 6 now removes the inflight tracker after synthesizing release notes, preventing accumulation across sessions.

## Maintenance

### Rename: create-claude-workspace → create-ulysses-workspace

The package and CLI have been renamed throughout `package.json`, `bin/`, `lib/`, `README.md`, docs, and template skill/recipe/rule examples. Historical release notes retain the old name — they describe what was shipped at the time.

**Action required for existing workspaces:** Run `npx create-ulysses-workspace --upgrade` from your local clone of this repo, then `/workspace-update` inside each workspace. The update will pick up the rename along with the other fixes.
