# v0.7.0 Release Notes

**Date:** 2026-04-13

Minor version bump — this release ships the new `/build-docs-site` skill, a comprehensive orchestrator for building Docusaurus documentation sites that consolidates scattered knowledge sources into a coherent branded reference with theme-aware diagrams.

## Features

### `/build-docs-site` — comprehensive documentation site builder

The workspace now ships a new skill that orchestrates the full lifecycle of building a Docusaurus documentation site for a project. The skill treats existing documentation as raw material to be replaced rather than augmented — bad docs become exponentially better when rewritten from scratch, and even good docs gain from a consolidated pass that folds in scattered knowledge sources (codebase, shared context, chat dumps, external exports) into one coherent reference.

The skill walks through eight explicit phases — framing, source gathering, spec writing, scaffolding, content writing, review, diagrams, and completion — and refuses to skip ahead. Each phase has checkpoints and a clear definition of done. Framing opens with eight sequential questions covering audience, existing documentation, external knowledge sources, research treatment, language constraints, brand identity, reading pattern (with chapter-length sizing derived from engagement style and cohesion preference), and structure. Migration is not a separate phase — the act of writing the new site **is** the migration, and Phase 3's mapping table plus Phase 6's coverage check verify that nothing was dropped on the way.

The skill was generalized from a real multi-session effort that produced 17 chapters and 26 diagrams for a separate project. Several hours of hard-won technique are captured and made reusable:

- **Theme-aware diagram primitives.** SVG diagrams that honor light/dark themes cannot use CSS variables in `fill="var(--x)"` or inline styles — Chromium's SVG pipeline does not paint those reliably. The working pattern is class-based selectors (`fill: var(--x)` rules applied via `className`), and the bundled primitives library (`Box`, `Arrow`, `SectionTitle`, `Region`, `DiagramContainer`, plus a `tokens` module) uses this approach by default.
- **Project-aware leak detection.** The skill's `leak-grep.mjs` builds its forbidden-term list from the project's actual dependency manifests (`package.json`, `pyproject.toml`, `Cargo.toml`, etc.) rather than a hardcoded master list. A React project's docs won't over-trigger on generic React terms, but a stray `Zustand` reference will still be caught — because Zustand is in `package.json`. The script is dependency-free Node so it runs anywhere the workspace runs.
- **Bulk fill migration.** `bulk-fill-migration.py` converts hand-written diagram components from `fill={colors.X}` to `className={cls.fill.X}` and handles three known regression cases by design: duplicate `className=` attributes, missing `cls` import, and variable-bound fills that need manual review.
- **Content writing rules.** Chapter-by-chapter on the main thread (no subagents — voice consistency matters more than parallelism for long-form prose), Chapter 1 written last (it's the hardest and benefits from knowing what comes after), periodic session tracker updates (the skill takes hours and context compaction is a real hazard), and four distinct review sweeps before declaring done.

### New skill directory convention

This is the first skill in the template with a sub-directory structure. `build-docs-site/` contains:

- `SKILL.md` — the eight-phase orchestration flow
- `templates/primitives/` — six React/SVG component files (Box, Arrow, SectionTitle, Region, DiagramContainer, tokens)
- `templates/custom.css.tmpl` — Infima overrides, `--dx-*` diagram variables, `.dx-fill-*`/`.dx-stroke-*` rule layer, typography polish (justified body, heading spacing, content column width)
- `templates/docusaurus.config.ts.tmpl` — `routeBasePath: '/'`, `blog: false`, brand placeholders
- `templates/sidebars.ts.tmpl` — clickable category link pattern
- `templates/spec.md.tmpl` — the design-docs-site.md template that Phase 3 fills in
- `scripts/leak-grep.mjs` — the dependency-derived leak detector
- `scripts/forbidden-word-grep.mjs` — a user-supplied wordlist sweep
- `scripts/bulk-fill-migration.py` — the fill→className migration helper
- `checklists/framing.md` — the eight framing questions with answer options and downstream effects
- `checklists/review.md` — the four review sweeps with commands and decision flows
- `checklists/pitfalls.md` — five captured pitfalls (CSS variables in SVG, bulk migration regressions, Playwright viewport screenshot quirks, arrowhead markers, Notion exports)

Every previous skill in the template is a single `SKILL.md` file. The convention break is justified here because the skill orchestrates substantial bundled tooling that would not fit in one file — primitives libraries, migration scripts, and multi-page checklists earn their own homes. Other complex skills (`workspace-init` at 464 lines, `complete-work` at 282 lines) could adopt the same pattern later if it proves out.

## Known Issues

A few items are deferred to v0.8.0 or later as the skill sees real use:

- **End-to-end testing against a real project.** The skill has not been run from Phase 0 through Phase 8 on a real target. The grep scripts were smoke-tested individually and produced expected output, but the full orchestration flow needs a real Docusaurus project, which doesn't exist in any of this workspace's repos yet. First real use will surface whatever the design missed.
- **Bulk fill migration smoke test.** The Python migration script handles three known regression cases by design but was not tested against a fixture. The first real use against a chapter components directory may surface edge cases in the JSX parsing.
- **Existing docs site handling.** If a user points the skill at an existing Docusaurus site, it currently scaffolds from scratch. A future enhancement could read the existing config and structure as a starting point. Not required for this release.
- **Chapter 1 inversion override.** The skill writes Chapter 1 last by default, following a common technical-writing convention. No user-facing override was added. If users prefer writing in order, they would need to manually skip the inversion.
