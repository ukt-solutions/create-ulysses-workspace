# v0.8.0 Release Notes

**Date:** 2026-04-13

Minor version bump. v0.8.0 restructures the on-disk layout so every work session lives in a single self-contained folder. The change is breaking on disk — existing workspaces need a one-time manual upgrade — but the user-facing skill API (`/start-work`, `/complete-work`, `/pause-work`, etc.) is unchanged.

## Features

### Work sessions live in one folder

Every work session now occupies a single folder at `work-sessions/{name}/`. Inside that folder: the workspace worktree (`workspace/`), nested project worktrees underneath it (`workspace/repos/{repo}/`), and a unified session tracker (`session.md`) plus any specs and plans for the session. Completing the session tears down the whole folder in one step — worktrees, tracker, specs, plans, all of it.

The previous layout scattered session state across three locations: project worktrees at `repos/{session}___wt-*/`, a JSON marker at `.claude-scratchpad/.work-session-*.json`, and a markdown tracker at `shared-context/{user}/inflight/session-*.md`. Specs and plans floated separately in the project worktree root. There was no clean unit of "this is one session, here is everything for it," and debugging a stuck session meant piecing context back together from all four locations. That's gone.

The old layout also relied on a `repos/` symlink inside every workspace worktree so nested path references would resolve. That symlink was the source of the v0.5.1 destructive gitignore bug, which is now patched defensively but remains a foot-gun by design. v0.8.0 removes the symlink entirely: the workspace worktree contains a **real** `repos/` directory with the project worktrees nested inside it, and the single `.gitignore` line `repos` (no trailing slash) covers both the workspace root's source clones and every worktree's nested `repos/`. One line, both uses, no symlink.

Source clones at `repos/{repo}/` at the workspace root still exist and still stay on their default branches. Clones are workspace-scoped state (one per repo, used by all sessions) and keeping them out of `work-sessions/` preserves the lifecycle distinction between "long-lived infrastructure" and "ephemeral session state."

### Unified session tracker

`work-sessions/{name}/session.md` replaces both the old JSON marker and the old markdown inflight tracker. One file, one source of truth. Machine state (status, branch, repos list, chat session log) lives in YAML frontmatter; human content (progress notes, captured reasoning from `/handoff` and `/braindump`, decisions) lives in the body. Scripts and hooks read and write the frontmatter through a new helper at `template/.claude/lib/session-frontmatter.mjs` that is lossless by construction — rewrites touch only the fields that actually changed, leaving every other byte of the file untouched.

The parser is hand-rolled rather than depending on `js-yaml`. It is scoped to the exact subset of YAML the workspace uses (flat scalars, flat lists, lists of mappings, inline lists) and has a 12-scenario unit test covering byte-identity preservation on no-op updates, ISO timestamp round-trips, nested `chatSessions` arrays of mappings, inline list edge cases, and quoting for strings containing colons or special characters. 77 assertions pass. The losslessness matters because hooks fire frequently — a naive parser that reordered fields or stripped comments on every rewrite would produce a git diff on every chat open.

The `/handoff` and `/braindump` skills, when run during an active session, now update the tracker body rather than writing to the deleted `shared-context/{user}/inflight/` directory. Outside an active session, they still create standalone files in `shared-context/{user}/` as before.

### Mandatory teardown order

Nested project worktrees introduced a subtle hazard that was verified empirically before the spec was written. If you remove a workspace worktree that contains nested project worktrees, git silently deletes the directory tree — including the nested worktrees' `.git` files — and leaves orphan worktree records in the project repos marked `prunable`. The removal appears to succeed (exit 0, directory gone) but the project repos are left with stale admin state that causes confusing errors on the next worktree operation.

The cleanup script (`cleanup-work-session.mjs`) enforces the safe order:

1. For each repo in the session tracker's `repos:` list, remove the nested project worktree first (`git -C repos/{repo} worktree remove ...`)
2. Then remove the workspace worktree
3. Then `git worktree prune` on each project repo as a belt-and-suspenders cleanup for orphans from any prior misuses
4. Delete local branches in all repos
5. `rm -rf work-sessions/{name}/` — the tracker, specs, plans, and any local-only artifacts vanish in one step

`/complete-work` documents the order explicitly in its Step 11 so future debugging surfaces it.

### Scratchpad rename

`.claude-scratchpad/` is renamed to `workspace-scratchpad/`. The dot-prefix was reflexive (modeled on `.claude/` without a reason to be hidden) and made the directory invisible in normal `ls` for no functional gain. The new name is visible and tells users immediately what it is. The `scratchpadDir` field in `workspace.json` configures the location, so anything that resolves the path through the config picks up the new default automatically. Upgrading workspaces can either rename in place (`mv .claude-scratchpad workspace-scratchpad`) or let the directory lazy-create and move any remaining content manually.

### Lazy directory creation

`repos/`, `work-sessions/`, and `workspace-scratchpad/` are all lazy-created on first use. The CLI scaffolder (`create-ulysses-workspace --init`) no longer pre-creates any of them, and all `.keep` files are deleted from the template. Scripts and hooks that need to write into these directories call `mkdirSync(..., { recursive: true })` before their first write. This removes three empty placeholder directories from every fresh scaffold and eliminates the `.keep` convention entirely.

### `/complete-work` handles non-GitHub remotes

Previously `/complete-work` hard-coded `gh pr create` and `gh pr merge` with no fallback. Sessions in workspaces using local bare remotes, mirrors, or non-GitHub hosts had to complete the merge manually.

v0.8.0 splits Step 7 into a remote-type detection phase that classifies each repo per session as GitHub, local-or-bare, other, or no-remote. Step 8 and Step 9 each have two variants: the GitHub PR flow (unchanged) and a local merge flow that pushes the feature branch, fast-forwards the default branch via `git push origin HEAD:{default-branch}`, deletes the feature branch from the remote, and pulls the source clone. Repos with no remote at all prompt the user to create one, provide a URL, or keep the work purely local. The unified summary in Step 9 adapts its wording — "Merge all locally? [Y/n]" instead of "PR #n" — when no PRs exist.

This was surfaced by the v0.8.0 smoke test on a throwaway workspace with a local bare-mirror origin and was fixed in the same session before merge.

### Gitignore pattern for tracked session files

The session tracker, specs, and plans inside each session folder need to be tracked so `git push` of the workspace branch carries them across machines. Everything else inside `work-sessions/{name}/` (worktrees and their contents) needs to stay ignored.

The naive approach — `work-sessions/` directory ignore plus `!work-sessions/*/session.md` negations — **silently fails**. Git cannot re-include files beneath an ignored parent directory, so the negation patterns have no effect when the parent is excluded as a directory entity. The first draft of this refactor had this bug. It was caught during the throwaway smoke test by running `git check-ignore -v work-sessions/foo/session.md` and discovering the file was silently dropped.

The working pattern is:

```gitignore
work-sessions/**
!work-sessions/*/
!work-sessions/*/session.md
!work-sessions/*/design-*.md
!work-sessions/*/plan-*.md
```

`work-sessions/**` ignores every descendant at every depth. `!work-sessions/*/` re-includes each session folder as a walkable directory. `!work-sessions/*/session.md` (and friends) re-includes the specific tracked files inside each session folder. Nested contents like `work-sessions/{name}/workspace/...` stay ignored because they are matched by `**` and never re-included. The gitignore file ships with a comment explaining the subtlety so future maintainers don't regress.

## Breaking changes

### Disk layout

The on-disk shape of a workspace changes substantially. Any skill, script, or tool that reached into `repos/{session}___wt-*/`, `.claude-scratchpad/.work-session-*.json`, or `shared-context/{user}/inflight/` needs to know about the new locations. All template-owned skills, scripts, and hooks are updated to use the new paths. Custom skills or external tooling outside the template need to be updated manually.

### Upgrading existing workspaces

Existing workspaces upgrade via a manual procedure documented in chapter 10 of the docs. There is no auto-migration and no migration skill — clean state is required before upgrading, and the migration is a one-time event per workspace.

The procedure:

1. Drain in-flight work. For each active or paused session, either `/complete-work` it (if shippable) or `/pause-work` and merge the resulting PR manually. Verify `ls repos/ | grep ___wt-` returns empty, `ls .claude-scratchpad/.work-session-*.json` returns empty, and `ls shared-context/{user}/inflight/` returns empty. Commit any pending workspace changes to main.
2. Pull the new template (`git pull` in the workspace repo) and run `/workspace-update` to apply the staged payload.
3. Manually clean up the old layout: `rm -rf .claude-scratchpad/` (replaced by `workspace-scratchpad/`), `rmdir shared-context/{user}/inflight/` if the directory still exists and is empty. Commit.
4. Smoke test by running `/start-work blank`, creating a trivial change inside `work-sessions/test-upgrade/workspace/`, and confirming `/complete-work` removes the whole folder cleanly.
5. For workspaces with multiple machines: pull on the second machine after upgrade, and the tracked `session.md`/`design-*.md`/`plan-*.md` files come along with the workspace branch automatically. Worktrees are local-only and recreate on first `/start-work` resume.

If you have an in-flight session that cannot be drained (mid-implementation, can't easily pause-and-ship), it is possible to migrate it in place by moving the worktree with `git worktree move`, writing a `session.md` tracker by hand using the new helper, and cleaning up the old-layout artifacts. This was demonstrated on the dogfood workspace's `ulysses-ui` session as part of the v0.8.0 cutover.
